# 임베딩과 벡터 검색 (Embeddings & Vector Search)

임베딩(embedding)은 텍스트를 **고차원 숫자 벡터**로 바꾼 표현이다. 의미가 비슷한 문장은
벡터 공간에서 가까이, 다른 의미는 멀리 위치한다. RAG 의 검색 단계는 이 "의미적 거리"를
이용해 질문과 가까운 문서 청크를 찾는다.

## 코사인 유사도 (Cosine Similarity)

두 벡터가 얼마나 비슷한지 재는 대표적 척도가 **코사인 유사도**다. 두 벡터 사이의 각도의
코사인 값으로, **1 에 가까우면 매우 유사**, 0 이면 무관, -1 이면 정반대를 뜻한다. 벡터의
길이(크기)가 아니라 **방향**만 보기 때문에, 문서 길이가 달라도 의미 유사도를 안정적으로 잰다.

검색은 보통 이렇게 동작한다: 질문을 임베딩한 벡터와, 저장된 모든 청크 벡터 사이의 코사인
유사도를 계산해 **가장 점수가 높은 상위 k개**를 반환한다. 이를 의미 검색(semantic search)
또는 최근접 이웃(nearest neighbor) 검색이라 부른다.

## 임베딩 모델

- OpenAI 의 `text-embedding-3-small` / `text-embedding-3-large` 같은 모델이 텍스트를
  벡터로 바꾼다. 작은 모델은 빠르고 저렴하며, 큰 모델은 정확도가 높다.
- 같은 임베딩 모델로 **문서와 질문을 모두** 임베딩해야 같은 공간에서 비교할 수 있다.

## 벡터 스토어 (Vector Store)

임베딩 벡터와 원문·메타데이터를 함께 저장하고 유사도 검색을 제공하는 저장소다.

- **InMemoryVectorStore**: 메모리에만 둔다. 가볍고 예제·프로토타입에 좋지만, 프로세스가
  끝나면 사라져 재시작 시 다시 임베딩해야 한다.
- **Chroma / FAISS / PostgreSQL(pgvector)**: 디스크·DB 에 영속 저장한다. 한 번 인덱싱하면
  재시작해도 재임베딩 비용이 들지 않아 대용량 코퍼스에 적합하다.

## 정리

질문과 문서를 같은 임베딩 공간에 올리고 코사인 유사도로 가까운 청크를 찾는 것이 벡터 검색의
핵심이다. 이 검색 결과를 LLM 에 근거로 넘기면 RAG 가 완성된다.
