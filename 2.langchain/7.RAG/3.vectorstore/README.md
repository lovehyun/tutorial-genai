# 3.vectorstore — ChromaDB 영속화 · 검색 · 컬렉션 배치 전략

벡터 DB(Chroma)에 **임베딩을 저장(영속화)** 하고, **어떻게 검색** 하며,
여러 문서를 **어떤 컬렉션 구조로 배치** 할지를 단계별로 다룹니다.

> 핵심 한 줄: `similarity_search` 한 번이 보는 범위 = **한 컬렉션**.
> 그래서 "어디에 담느냐(분리/통합/혼합)"가 "어떻게 검색되느냐"를 결정합니다.

## 학습 흐름 (5그룹)

```
[기초·영속화]      3.1 txt → 3.2 pdf → 3.3 들여다보기
[검색 모드]        3.4 similarity / with_score / MMR
[배치 전략] ★핵심  3.5 분리(sep) → 3.6 통합(unified) → 3.7 혼합(mixed)
[검색 동작]        3.8 통합 vs 분리 검색 차이 → 3.9 정답이 두 문서에 쪼개진 경우
[부록]            3.10 FAISS 와 비교
```

## 분리 vs 통합 vs 혼합 — 한 장 요약

```
            ── 적재(어디에 담나) ──            ── 검색(어떻게 찾나) ──
3.5 분리     [컬A: nvme][컬B: ssd]              각 컬렉션 따로 / merge 필요
3.6 통합     [컬A: nvme+ssd+pdf]                한 번에 다 검색 (전역 top-k)
3.7 혼합     [컬A: txt들][컬B: pdf1][컬C: pdf2]  묶은 건 같이, 나눈 건 따로
                        │
3.8 검색 ───────────────┘  위처럼 담았을 때 실제 검색 동작 (top-k 통합/각각, filter, 재정렬)
3.9 교차    "정답이 두 문서에 쪼개진" 질문도 통합이면 한 번에 종합됨
```

- **언제 통합(3.6)?** 평소 같이 검색하고 싶을 때 = 대부분의 실무 RAG. 특정 파일만 필요하면 `filter`.
- **언제 분리(3.5)?** 사용자/테넌트별 **격리**가 필요할 때 (A 의 문서가 B 검색에 절대 안 나와야 함).
- **순서 팁:** 통합이 더 쉽고 기본 → 분리는 "통합하려면 merge·재정렬이 왜 필요한가"를 보여주는 특수 케이스.

## ★ 가장 중요한 관전 포인트 — top-k 가 "각각" vs "합쳐서"

같은 질문이라도 store 를 어떻게 두느냐에 따라 **top-k 가 적용되는 단위**가 달라집니다.

### (a) 따로 질문 / store 분리 — top-k 가 컬렉션 '각각' 에 적용

```
  질문 "A 와 B 를 둘 다?"
        │
        ├──► [Store A] ──유사도──► A 안에서 top-2 ─┐
        │                                          ├──► merge ──► A 2개 + B 2개
        └──► [Store B] ──유사도──► B 안에서 top-2 ─┘              (양쪽 무조건 포함)

   각 컬렉션이 자기 몫 k개를 보장 → A·B 가 반드시 둘 다 대표됨.
   단점: 합치는 merge 코드를 직접 짜야 함 (3.5 / 3.8 의 2-b·2-c).
```

### (b) 합쳐서 질문 / store 통합 — top-k 가 '전체 풀' 에 한 번 적용

```
  질문 "A 와 B 를 둘 다?"
        │
        └──► [Store  A + B  하나의 후보 풀] ──유사도──► 전체에서 top-4

     거리(작을수록 가까움) 예시:
        A1 0.21   A2 0.24   A3 0.26   A4 0.29   │   B1 0.31   B2 0.40
        └──────────────  상위 4개  ─────────────┘
        top-4 = A1 A2 A3 A4         ← A 가 점수를 독식
        B 는 한 개도 못 들어옴  ✗

     → "A 와 B 를 합쳐야 풀리는 질문" 인데 B 청크가 누락 → 반쪽 답 / 답 못함
```

### 그래서 — 통합의 함정과 대처

| | 분리 (각각 top-k) | 통합 (합쳐서 top-k) |
|---|---|---|
| 양쪽 소스 대표성 | **구조적으로 보장** (각 k개씩) | 점수순이라 **한쪽이 독식하면 누락 가능** |
| 합치는 코드 | 필요 (merge) | 불필요 (한 번에) |
| A+B 종합 질문 | 유리 | k 가 작거나 한쪽이 강하면 **취약** |
| 일반 단일 질문 | 과함 | 간단·기본 |

- **통합 약점 완화법:** ① `k` 를 넉넉히(→ B 도 들어올 자리, 3.9) ② **MMR**로 다양성 확보(3.4) ③ 소스별로 나눠 뽑아 합치기(= 사실상 분리 방식)
- **결론:** "A 절반 + B 절반" 종합형 질문이 잦다면 → **통합 + 큰 k + MMR**, 또는 아예 **분리 + 소스별 top-k** 가 더 안전. 통합이 항상 정답은 아님(점수 독식 함정).

## 실행

```bash
cd "2.langchain/7.RAG/3.vectorstore"
python 3.1_persist.py          # 처음엔 ./chroma_db 생성, 다시 실행하면 로드만
python 3.6_single_collection_unified.py
```

> - 모든 예제는 `../DATA/` 의 파일을 사용하고, 임베딩에 OpenAI 키(`.env`)가 필요합니다.
> - 파일마다 **컬렉션 이름이 달라서** 같은 `./chroma_db` 안에 공존합니다 (서로 안 섞임).
> - 3.9 는 임시 컬렉션을 쓰고 끝에 `delete_collection()` 으로 정리합니다.
